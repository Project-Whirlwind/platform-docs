### `comm-gateway`
**Purpose:** Single integration point for all inbound and outbound communication. Abstracts Twilio (SMS) and Mailgun (email) behind a provider-agnostic API. Enforces configurable safety limits on all inbound content. Products never call Twilio or Mailgun directly.
**Repo:** `github.com/project-whirlwind/comm-gateway`
**Status:** Planned
**Stack:** Elixir / Phoenix, Oban, PostgreSQL
**Exposes:**
- `POST /v1/messages/sms` — send an SMS via configured provider
- `POST /v1/messages/email` — send an email via configured provider
- `POST /v1/webhooks/twilio` — receive Twilio inbound SMS webhooks
- `POST /v1/webhooks/mailgun` — receive Mailgun inbound email webhooks
- `POST /v1/webhooks/twilio/status` — Twilio delivery status callbacks
- `POST /v1/webhooks/mailgun/status` — Mailgun delivery status callbacks
**Key behaviours:**
- Webhook handlers always respond immediately (< 15ms) — never block on downstream processing
- Signature validation is the first step — invalid requests never reach limit checks or parsing
- Safety limits enforced before forwarding (size, link count, rate, recipient count)
- Truncated events carry `truncated: true` and `link_count_original` for product transparency
- Provider adapters implement a shared behaviour — SMS and email providers are swappable via config
**API contract:** `api-contracts/comm-gateway/v1/openapi.yml`
**Full spec:** `services/comm-gateway.md`

### `ai-gateway`
**Purpose:** Provider-agnostic AI completion layer. Routes requests to Claude, GPT-4, or other LLMs. Manages conversation state and threading. Enforces rate limits per caller.
**Repo:** `github.com/project-whirlwind/ai-gateway`
**Status:** Planned
**Stack:** Elixir / Phoenix, LiteLLM, PostgreSQL, Oban
**Exposes:**
- `POST /chat` — send a completion request with optional conversation context
- `GET /conversations/:id` — retrieve conversation history
- `DELETE /conversations/:id` — clear a conversation thread
**API contract:** `api-contracts/ai-gateway/v1/openapi.yml`

---

### `revenue-ledger`
**Purpose:** Double-entry revenue tracking using TigerBeetle. Tracks revenue generated by and attributable to users across all products. Foundation for future profit-sharing and co-op distribution. Runs as a background proof-of-concept initially.
**Repo:** `github.com/project-whirlwind/revenue-ledger`
**Status:** Planned (background POC)
**Stack:** Elixir / Phoenix, TigerBeetle
**Exposes:**
- `POST /accounts` — create a revenue account for a user
- `POST /transfers` — record a revenue transfer/attribution
- `GET /accounts/:id/balance` — get account balance
- `GET /accounts/:id/transfers` — get transfer history
**API contract:** `api-contracts/revenue-ledger/v1/openapi.yml`
